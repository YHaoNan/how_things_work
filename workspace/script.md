# 它们是什么
假设你是一名游戏开发者，你正在指定的游戏机平台上开发游戏，你希望你的游戏以4K分辨率运行，可在测试过程中，你发现在这个硬件上，1080P输出已是极限，再提升分辨率，帧率便会明显的下降。因为画面越清晰，需要计算的像素点越多，4K要计算的像素点是1080P的四倍。

> 此时，我想搭配这样的动画：
>
> 一个曲线chart，从左滑到右侧。纵坐标是分辨率，横坐标是帧率，开始在720p，帧率是60fps，随着分辨率的提升，帧率会下降，1080p到达30fps，直到4K，帧率是10fps。同时，我希望你能够在chart右侧添加一个弹跳的小球示例，分辨率低下时，小球可以有很多毛边，甚至可以用像素画来模拟，但是此时小球运动的帧率很高，随着曲线的滑动，小球变得越来越清晰，但其帧率越来越低。在小球上方，我希望能够展示当前的帧率和分辨率，并且在720P、1080P和4K时有明显停顿。

动动脑筋，如何在保证高分辨率的情况下，保持小球运动帧率在可接受范围内？如在4K分辨率下，保住30fps的帧率？

我们立刻就能想到一个作假的办法，我们每帧只选取1/4的像素点进行渲染，这样我们便可以让渲染4K分辨率的计算量与渲染1080P的计算量相同，在快速交替的情况下，人眼可能难以察觉这一差异。这种经典的简单算法叫做棋盘算法，相比于我们今天要介绍的技术，它要简单朴素的多，但它却是一个非常好的讲解其它技术的基础。

> 此时，我想搭配这样的动画以解释棋盘算法：
>
> 展示一个矩形，其内部被划成一个一个的小正方形，观感类似棋盘，以及一个进度条，进度条有4步，每一步，棋盘中的1/4个格子高亮展示，表示它们是当前被激活的像素点。

看过了棋盘算法，今天我们要介绍的主角则是采取另外的思路，它们接收一个低分辨率的图像，然后使用某种计算量可控的算法将其提升为高分辨率，最终渲染输出。它们被统称为“超分（辨率）技术”。
