# 前言：如何从Next-Token Prediction到自动写代码
在大语言模型技术如此泛滥的今天，我想大家应该多少都对其内部的运行原理有了一定的了解（不论是因为好奇还是焦虑）。所以大家一定都知道，大语言模型就是一个进行 next-token prediction——也就是下一个词元预测 的，巨大的概率模型，即：你给他一段话，他为你预测出下一个单词，模型预测出下一个词元后，我们会把这个新词元加回到输入中，让模型继续预测下一个，如此循环，就能生成一段话。

“在”字闪烁后，从模型顶端输入整个句子。模型内部有光点沿连接线流动，最终从底部输出一个字“哪”。  
“哪”字被添加到原句末尾，新句子“迪迦奥特曼昨天在哪”重新输入模型。重复此过程，每次生成的字都高亮。  
模型依次输出“个”、“频”、“道”、“播”、“出”、“？”，最终形成完整句子：“迪迦奥特曼昨天在哪个频道播出？”

从下一个单词预测，到生成软件项目，这个跨度大到让人不敢置信。

但实际上要构建一个真正的产品，我们还忽略了如下复杂性：

1. 如何让聊天模型具有记忆？这一般通过软件构建，我们会将完整的或者压缩过的聊天轨迹每次都重新发给模型
2. 如何获取那些用于训练的对话文本？常见的方式有基于其它的模型蒸馏、基于通用数据集、或者引导AI自己生成类似的文本等
3. 在聊天场景下如何限制模型的行为，如不让它说政策、道德、伦理相关的内容，如何调节它的偏好，如它要讨好用户而非辱骂用户。这个过程通常被称为指令微调或Alignment，相关研究有InstructGPT、Self-Instruct等
4. 在向指定方向特调模型时，模型通常会在其它能力上减弱，或直接丧失其它能力，如何降低这方面的影响？相关研究方向有混合训练Mixture of Tasks，让模型用自己重新组织语言、或强调数据质量的Less Is More for Alignment等。

## 获得接触外部世界的能力
直至现在，模型已经学会了聊天和思考，但它还有一个致命的问题，即它无法观察到外部环境，这会带来种种不便：

1. 模型了解的所有知识限于它训练完成时人类产生的所有资料，新的资料他不知道
2. 很多复杂问题无法通过通用知识解决，如果让模型帮你写代码，它至少能读写你的项目



于是，ReAct等技术被发明出来，它大概是以这样的提示词和模型交互：



> 生成动画，展示ReAct的工作流程。
>
> 画面左侧是一个对话框，显示人类提问：“请帮我编写一个小程序输出爱心...”。  
画面右侧是一个代表“AI模型”的方块。  
从AI模型方块中，首先弹出气泡“**思考**：我要编写一个程序，所以我要使用`write_file('./heart.py', '程序内容')`”。  
紧接着弹出第二个气泡“**行动**：`{"action": "write_file", "params": {"path": "./heart.py", "content": "..."}`”。  
画面底部出现一个代表“框架/环境”的长条区域。长条区域接收行动JSON，模拟调用工具，然后返回一个结果气泡“**观察**：`write_file`返回了OK”。  
观察气泡被连线送回给AI模型方块。AI模型方块再次弹出新的“思考”气泡，形成一个循环箭头，表示持续迭代。
>



人类：

+ 请帮我编写一个小程序输出爱心，如下是你能使用的工具：write_file(path, content)，请你以思考，行动，观察的方式完成任务

模型：

+ **思考**：我要编写一个程序，所以我要使用`write_file('./heart.py', '程序内容')`
+ **行动**：`{"action": "write_file", "params": {"name": "path", "value": "./heart.py", }`...

框架：

+ 解析模型返回的标准行动json，调用对应工具
+ 通知模型调用结果

模型：

+ **观察**：观测到了`write_file`返回了OK，任务已经执行成功



这就是老生常谈的Agent技术，相对于前面两种技术，它具有如下特性：

1. 它必须基于多轮对话，构建思考、行动、观察、思考、行动、观察的轨迹循环，直到问题解决
2. 它必须有一个代码框架驱动，框架用于提供工具、帮助调用工具



类似聊天特调和思考特调，现在各家模型厂商也会将这种思考、行动、观察轨迹循环内化到模型内部，agent能力也是当下主流模型的必要能力。



> 扩展“特调训练”动画（第二个动画）。
>
> 在“对话文本”标签之后，新一波的“思考链条”标签（内容如“**思考**：用户想要… **行动**：… **观察**：…”）飞向模型方块。  
模型方块在吸收这些标签后，其内部的网络连接线变得更加粗壮和明亮
>



看起来，有了Agent，大模型终于能实现从Next-Token Prediction到Vibe Coding的跨越了。但现实是，构建真正的产品依然有很多复杂性：

1. Agent应该如何真正和模型交互？ReAct，Focused ReAct是这一领域的一些研究。另外，LangChain等框架的Agent实现也是值得研究的方向。
2. 现实世界的任务轨迹并不都是实例中那样简单，比如模拟人类操作浏览器的任务具有天然的人类操作延迟，如何描述这些复杂的轨迹是一个问题。并且构建通用的格式对于训练和最终效果都有巨大的好处。AgentTuning/ADP等是这方面比较新的研究。
3. 相比于聊天、思考，任务的轨迹收集要困难得多，这直接导致了构建可靠训练数据的复杂度飙升，Nex-N1是这方面较为前沿的研究，它通过构建三个组件，将复杂结构多agent生成、自动化执行器构建、以及自动化轨迹质量校验、整合构建成完善的工作流，可以自动化的生成高质量的轨迹。
