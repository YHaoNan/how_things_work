# Your work
1. 规划：分析`script.md`，规划要生成哪些动画，在项目路径中生成.animation-plan.md文件
    1. 一些动画在script.md中已有用户提示，此时应严格按照用户的提示生成
    2. 你需要将几乎所有脚本中可以以动画形式可视化的地方都生成动画，让用户能直观的从视觉上理解所讲内容
    3. `.animation-plan`应该是markdown格式，其中每一个动画是一个段落，包含：
       1. 动画概述：包含动画的简要概述
       2. 具体内容设计：
          1. 步骤1：步骤1概述
             1. UI设计：描述步骤1的全部UI设计细节
             2. 动画设计：描述步骤1的全部动画设计细节
          2. 步骤2：步骤2概述
             1. UI设计：描述步骤2的全部UI设计细节
             2. 动画设计：描述步骤2的全部动画设计细节
    4. `.animation-plan`应使用中文
2. 逐个生成`.animation-plan`中的动画，每次仅生成一个，生成后停下与用户交互。针对这个动画，用户可能会通过多轮对话不断与你进行沟通微调，直到用户满意，你再生成下一个，并且在此过程中你需要将沟通出的新内容回写到`.animation-plan`。同时，用户也可能向`.animation-plan`中新增或删除动画。

!!! 不要一次性生成全部动画！不要一次性生成全部动画！！！！！！

# Animation Generation Step By Step Instruction(For single conversation turn)

1. 明确要生成的是哪个动画（来自`.animation-plan`）
2. 将动画拆解成多个演示步骤，如一个完整的演示动画会从A状态到B状态再到C状态，每一个状态转换都是一个步骤
3. 在on_build_ui中搭建UI：布局自适应，文本可换行，避免溢出裁切
4. 重要：在on_play中编排动画和时间线：步骤之间用yield* waitUntil('事件名')停顿（必须），这样做会在页面上新增一个event，我们便可以操作每一个步骤停留的时间以和文本匹配
5. 优先使用项目assets中的图片/视频资源，保证比例与清晰度
6. 当你的动画生成完成，你需要在画面稳定的时刻（如一个状态转换完成）添加测试点，用于检查当前画面的UI是否与你预期的有出入，或是有明显的错误，详见test-animation

22→当你遇到问题，你应该回到收集信息的步骤中
